## 0. 全局约束（每个任务都必须遵守）

1. **最小改动**。不新造抽象、不顺手重构、不做"未来防御"。拿不准就停下来问，不要自行扩大范围。
2. **Swift 规矩**：Swift Testing（`@Test`/`#expect`，禁 XCTest）；无 ViewModels；禁 `onAppear { Task {} }`（用 `.task`）；禁 `.onChange(of:perform:)` 旧签名；UI 用 `@MainActor`，共享状态用 `actor`；避免 `@unchecked Sendable`。
3. **TDD**：先写失败测试再实现。碰 `LocalStorage` resettable keys / 全局 `UserDefaults.standard` 的测试必须包 `await SharedPersistenceTestLock.shared.withLock { ... }`。
4. **BLE 线格式纪律**：任何 wire 字节/payload 改动 → 同步改 `docs/BLE通信协议规格文档.md`（版本号递增 + 修订历史行）→ 同步飞书（走 `lark-doc` skill，文件↔token 映射已在其中）。改 `BLEDataEncoder.encodeX` 必须同步测试层镜像解码器 `parseX`（`BLEProtocolSimulationSupport.swift` 严格 `requireEnd()`，多一个空串都会红）。跑**全量** `swift test`，不要只跑 `BLEProtocolTests`。
5. **UI 改动必须模拟器目测**（build_run_sim + 截图），不目测不算完成。
6. **所有用户可见文案一律英文**（宠物对话/按钮/横幅/E-ink 文本）。中文只允许出现在代码注释。
7. **每个任务独立 commit**（conventional commits：feat/fix/docs/…），build + 全量 test 通过后才 commit，不堆批。
8. **BLE 字节是方向双命名空间**：出站 `Core/BLE/BLEProtocol.swift`（`BLEDataType`），入站 `Models/EventLog.swift`（`EventLogType.rawByte`）。同值不同向不同义是设计，不是冲突。

## 任务 1（P0）：专注打断判定重做 —— 当前实现与客户设计**相反**

### 背景与设计意图（客户 2026-07-09 确认）

- **设计**：硬件发来进入专注信号后，App 应显示**固定的专注模式界面**（App 在前台立即切换；在后台则用户下次打开 App 直接落在专注界面）。**打开/停留在 Kirole 专注界面不算打断**；**专注期间使用其它 App 才算打断**。
- **现状（全部已核实）**：
  1. 全代码库**唯一**打断记录点 = `ScreenActivityTracker.handleDidBecomeActive`（`FocusSessionService.swift:784-789`，监听 `UIApplication.didBecomeActiveNotification`，无过滤）→ **打开 Kirole 本身被记为打断**，与设计相反。
  2. 「使用其它 App」**完全无检测**：全仓库 grep `DeviceActivity` 零命中。会话在后台开始时（硬件发起的主场景），解锁手机用微信一小时不记打断，回到 Kirole 那刻才记一笔——方向反了。
  3. **正式专注界面不存在**：`FocusPetView` 仅被 Settings 的测试浮层 `FocusTestOverlayView` 使用（`SettingsFocusSection.swift:280-330`，带 "Fast-forward test" 开关的开发页）；`BLEEventHandler.handleEnterTaskIn` 无任何导航代码；`selectedTab` 只在 `AppHeaderView.swift:108/123/140` 用户点击处赋值。
  4. 深度专注（`FocusGuardService` / ManagedSettings shield）只**拦截**选定 App，**不记打断**，不能替代判定。

### 必须保持不变（这些与设计一致，勿动）

- 每满 30 分钟不间断 = 1 能量瓶（`FocusEnergyCalculator.bottlesEarned = minutes/30`，`FocusEnergyCalculator.swift:4-6`）。
- 打断时不足 30 分钟零头**作废、不跨段合并**；已得瓶**不回收**（`FocusTimeCalculator.countableBottles`，文件内注释已写明这是 spec）。
- 专注时长统计只计 ≥30 分钟的不间断整段（`countableFocusTime`：`duration >= threshold` 否则记 0）。
- 结束时按时间戳倒算（`endSession`，`FocusSessionService.swift:191-232`）——App 后台/锁屏不影响准确性，这个结算架构保留。
- **离线 replay 的 `enterTaskIn` 不补开专注会话**（`guard !isReplay`）——历史决定，有意为之，别"修"。
- 硬件触发链路本身：`0x10 enterTaskIn` → `BLEEventHandler.handleFocusSessionEvent` → `startSession`，以及 `0x11/0x12` 结束——不变。

### 平台约束（写给执行 agent，防止走错路）

- iOS **不允许** App 从后台把自己拉到前台。"自动显示专注界面"只能做到：(a) App 在前台时立即切换；(b) App 在后台时计时照常，**用户下次打开 App 时**自动落在专注界面（会话进行中为前提）。
- iOS **不允许**直接观察"别的 App 在不在前台"。检测"使用其它 App"的正路是 **Screen Time / DeviceActivity**（`DeviceActivityMonitor` extension + `FamilyActivitySelection`），与现有深度专注的 FamilyControls 授权同体系。注意 DeviceActivity 的事件粒度是"累计使用达阈值"回调（最小分钟级），不是秒级前台切换通知——验收标准按分钟级容忍度定。

### 实施要点

1. **新判定源与旧信号必须同批切换**：如果只删掉 didBecomeActive 打断而不接入新检测，会变成"永不打断、能量瓶白送"，比现状更糟。禁止拆半落地。
2. **正式专注界面**（新 UI，英文文案）：会话进行中打开 App 即见；展示任务名、经过分钟、当前分段进度/能量瓶、阶段（idle/warmup/building/deep 文案已存在于 `FocusPhase.swift`）。**Pet 页布局是客户钦定设计（CLAUDE.md），不许动它来腾地方**——专注界面是独立呈现（如 fullScreenCover；记得 `injectAppEnvironment()`）。
3. **打断语义全链路影响面**（改判定源时逐个过）：
   - `ScreenUnlockEvent` 的语义（从"Kirole 回前台"变为"检测到其它 App 使用"）；
   - `FocusTimeCalculator.countableBottles / countableFocusTime / currentSegmentStart` 的输入仍是同一结构，算法不动；
   - `onInterruptionRecorded` → `syncFocusHardwareDisplay`（0x14 实时推送归零画面）触发点跟随新信号；
   - 崩溃恢复持久化 `persistActiveSessionWithInterruptions`（打断即持久化）跟随新信号；
   - 当天统计 `interruptionCount` 口径说明。
4. **协议文档**：wire 格式（0x14 focus-status payload）**不变**，但 §focus-status 行为描述里"打断"的定义要更新 → 版本号 + 修订历史 + 飞书同步。
5. **测试**：`FocusTimeCalculator` 既有测试全保；新增判定源的单测（模拟 DeviceActivity 回调 → 打断记录 → 0x14 推送/持久化被调用）；`FocusSessionServiceTests` 等既有套件必须全绿。

### 已拍板决策（产品 2026-07-10 确认，执行 agent 按此实施、不再询问）

- **D-1 监测范围**：复用深度专注的**用户自选分心 App 清单**（即 `FocusGuardService.currentSelection()` 使用的那套 `FamilyActivitySelection`）做 DeviceActivity 监测——**不做**全类别监测。含义：打断检测只对用户勾选过的 App 生效；清单为空 = 等同检测未开启，走 D-2 的明示态。
- **D-2 授权与降级**：打断检测需要屏幕使用时间授权。未授权、授权失效、或分心清单为空时：**无打断检测 + UI 明示**（英文文案，示意 "Interruption detection is off — allow Screen Time access and pick your distracting apps to enable it"，具体措辞实现时定稿）。**禁止静默回退到旧的 didBecomeActive 信号**——旧打断路径在本次重构中彻底移除，不保留任何代码分支。
- **D-3 shield 交互**：深度专注下用户尝试打开被屏蔽的 App（被拦截页挡下、未实际使用）**不算打断**。实现注意：shield 拦截页的短暂出现不应累计出 DeviceActivity 使用阈值事件；若实测会误触发，必须过滤掉。

## 任务 2（P1）：IP/伙伴切换改为立即推送（当前最长延迟 1h/4h）

### 现状（已核实）

- `selectBuiltInCompanion`（`AppState+CustomCompanions.swift:104-117`）只调 `requestBLESync(reason:)` → 1.5s 防抖 → `performSync()` **不带 force**（`AppState+Sync.swift:367-377`）→ 被 `BLESyncPolicy.shouldSync` 卡住（`BLESyncPolicy.swift:12-17`）。
- 且 **character 字段被有意排除在 DayPack 指纹外**（`AppState+CustomCompanions.swift:113-115` 注释，2026-07-04 审计 F2 的决定）→ 切换不算"内容变化"→ 不触发提前同步 → 硬件最长停旧 IP 到下一个 1h/4h 窗口。
- `PetStatus(0x01)` 目前只在 `BLESyncCoordinator.performSync` 里发（约 `:129-132`）。
- 对照：自定义头像分支 `selectCustomCompanion` 已经是"立即推 `0x15`"（`:131-135`），有现成模式可抄。

### 改法（最小）

在 `selectBuiltInCompanion` 里（`updateUserProfile` 之后）**立即单发一帧 `0x01 PetStatus`**——仿照 `selectCustomCompanion` 的立即推送写法：`Task { @MainActor in try await BLEService.shared.sendPetStatus(...) }`，do-catch + `ErrorReporter.log`（本代码库不鼓励 `try?`）。

- **不要**把 character 纳入 DayPack 指纹（会连带整包重发，浪费；指纹排除的决定保留）。
- **不要**改成 `performSync(force: true)`（整轮同步太重，一帧 0x01 足够）。
- 断连时：发送失败记日志即可，不做重试队列——下一轮常规同步本来就会带上最新 characterId。
- 连点切换：每次选择各发一小帧，`BLERateLimiter.acquireWritePermit`（20 写/秒闸）天然兜底，无需额外防抖。
- `selectCustomCompanion` 分支顺手对齐：它目前也只请求节流同步 + 推 0x15 像素帧，但 0x01 里的 characterId 仍是旧内置名——是否同步单发一帧 0x01 由实现者判断（协议上 0x01 的 characterId 在自定义激活期间本就仅反映"最后一次内置选择"，见 `UserProfile.swift:18-20` 注释；不强求）。

### 验收

1. 模拟器/真机：切换内置伙伴后 **≤2 秒**内 BLE 日志可见 0x01 出帧（含新 characterId）。
2. 全量 `swift test` 绿。
3. `docs/BLE通信协议规格文档.md` §PetStatus 的"发送时机"描述追加"伙伴切换时立即单发"→ 版本递增 + 修订历史 + 飞书同步。
4. **依赖硬件确认（写进给硬件的沟通，不阻塞 App 实现）**：固件收到 0x01 后是否立即重绘形象。App 侧发了、固件不重绘的话，最终体验仍不达标——验收里"设备立即换形象"一条以硬件确认为准。

## 任务 3（P2，先验证后修）：专注期间 0x14 实时推送的后台可靠性

### 疑点（本次分析发现，尚无实锤）

`startFocusDisplaySyncLoop`（`FocusSessionService.swift:116-127`）是裸 `Task.sleep` 自循环，无 BGTask 兜底。App 有 `bluetooth-central` 后台模式（`Config/Info.plist:82-87`），蓝牙连着时进程大概率存活、循环大概率继续跑，但苹果对"后台裸定时器长时间运行"无强保证——30-90 分钟锁屏会话中途停摆的可能性无法仅凭代码排除。

### 行动（顺序执行，不许跳）

1. **先真机验证**：完整锁屏专注会话（≥60 分钟不碰手机），抓 BLE 日志确认 0x14 是否按能量瓶节奏持续到达设备。
2. 若验证**正常** → 关单，在本文件此节标注"已验证无问题+日期"。
3. 若确认停摆 → 再设计加固（候选：设备侧 0x30 唤醒时已有补推兜底，可评估是否够用；不预先焊死方案，按最小改动来）。
4. **注意**：若任务 1 重构触及该循环，合并到任务 1 一起处理，别做两遍。
