# AI 生成砖块/建筑 — 可行性评估与建议

> 日期：2026-03-07
> 状态：评估阶段，待讨论

## Context

客户建议：根据每天的任务完成和专注情况，AI 生成视觉元素（路灯、房子等），形成一个持续生长的像素小镇。因为 AI 开放设定，有更多可能性，不容易疲倦，带随机性。

需要评估：技术可行性、实现方案、难点、成本预算。

---

## 一、结论：可行，推荐"AI 决策 + 固件资产库"方案

可行性评分：**8/10**（在合理控制范围内完全可实现）

---

## 二、当前架构约束（决定方案选择的关键因素）

| 约束 | 现状 | 影响 |
|------|------|------|
| BLE 带宽 | DayPack 200-500 bytes，每日同步 ~17 次 | **不能传输图像像素数据**（120KB-192KB 一帧） |
| E-ink 刷新 | 全刷 12-36 秒，6 色 4bpp | 不适合频繁动态内容 |
| 设备渲染 | 固件本地渲染预置资产（宠物已是这个模式） | **必须走"参数指令 → 固件渲染"路径** |
| AI 服务 | OpenRouter/GPT-5.1，已有 CompanionTextService | 可复用现有 AI 管道 |
| 用户数据 | BehaviorAnalyzer 已有完成率/专注时长/连续天数 | 可直接作为 AI 输入 |

## 三、推荐方案：AI Village（像素小镇）

### 核心思路

```
用户行为数据 → AI 决策引擎 → 建筑指令(~20 bytes) → BLE → 固件渲染 → E-ink 像素小镇
```

### 运作流程

1. **每日结算时**，`BehaviorAnalyzer` 汇总当天数据：
   - 任务完成率、专注总时长、连续天数、最长单次专注、中断次数

2. **AI 生成建筑决策**（复用 OpenAIService，新增一个 prompt）：
   - 输入：用户行为摘要 + 当前小镇已有建筑列表 + 随机种子
   - 输出：结构化 JSON → `BuildingBlock`

3. **BLE 发送建筑指令**（仅 ~15-25 bytes）：
   ```
   buildingType:  UInt8   // 路灯=1, 小屋=2, 树=3, 长椅=4, 喷泉=5, 花园=6, 商店=7, 塔楼=8...
   variant:       UInt8   // 同类型的样式变体（最多 4-8 种）
   size:          UInt8   // S=1, M=2, L=3（由完成度决定）
   colorAccent:   UInt8   // 6色中的强调色
   gridPosition:  UInt8   // 小镇网格位置（0-24 for 5x5 grid）
   dayNumber:     UInt16  // 第几天（用于回溯）
   ```

4. **固件侧**：维护一个 5x5 或 6x4 的网格小镇，每个格子可放置一个建筑。新建筑出现时整屏重绘。

### AI 决策逻辑示例

| 用户表现 | AI 可能生成 | 大小 |
|----------|------------|------|
| 完成率 >80% + 专注 >2h | 房子、塔楼、商店（大型） | L |
| 完成率 50-80% | 路灯、长椅、花园（中型） | M |
| 完成率 <50% 但有专注 | 树、花盆、信箱（小型） | S |
| 连续 7 天完成 | 特殊建筑（喷泉、雕像） | L |
| 零完成但打开了 App | 杂草、落叶（有趣但低调） | S |

### 随机性来源
- AI prompt 中注入 `temperature: 0.9` + 随机种子
- 同类型有 4-8 种变体（如房子有茅草屋/砖房/木屋/现代屋）
- 颜色强调随机选取
- AI 可以根据"小镇整体氛围"做创意决策（避免重复）

## 六、难点分析

### 难点 1：固件像素资产制作（高）
- **问题**：需要大量 6 色像素风格的建筑小图
- **解决**：
  - 方案 A：请像素画师批量制作（质量高，成本 ~$500-1500）
  - 方案 B：用 AI 图像生成（DALL-E/Midjourney）+ 手动 6 色量化（质量中等，成本低）
  - 方案 C：程序化像素图（代码生成简单几何建筑，质量一般但零成本）
  - **推荐 B+C 混合**：核心建筑 AI 生成+手调，简单元素程序化

### 难点 2：小镇布局美观性（中）
- **问题**：随机放置可能视觉混乱
- **解决**：网格系统 + 规则约束（如大建筑不相邻、道路连通性检查）

### 难点 3：固件渲染内存（中）
- **问题**：ESP32-S3 PSRAM 2-4MB，需管理多个建筑资产
- **解决**：建筑小图用 RLE 压缩存储，按需加载到帧缓冲

### 难点 4：长期不单调（低-中）
- **问题**：建筑类型有限，几周后可能重复
- **解决**：
  - 季节/天气主题变化（春天樱花树、冬天雪屋）
  - 特殊事件建筑（连续 30 天 = 城堡）
  - 后续 OTA 固件更新追加新建筑包

## 七、成本预算估算

### 开发成本（一次性）

| 项目 | 预估 | 说明 |
|------|------|------|
| iOS 端开发 | 2-3 天 | 模型+服务+BLE编码+预览UI |
| 固件端开发 | 3-5 天 | 渲染引擎+BLE解析+资产管理 |
| 像素资产制作 | $500-1500 或 3-5 天内部 | 取决于方案选择 |
| 联调测试 | 2-3 天 | iOS <-> 固件 BLE 通信验证 |
| **合计** | **7-16 天 + $0-1500 资产费** | |

### 运营成本（持续）

| 项目 | 单价 | 日用量 | 月成本 |
|------|------|--------|--------|
| AI 建筑决策 | ~200 tokens/次 | 1次/天/用户 | ~$0.003/用户/月 |
| 现有 AI 文案 | 已有成本 | - | ~$0.06/用户/月 |
| **AI 总增量** | - | - | **+$0.003/用户/月（可忽略）** |

### 成本结论
- AI 调用成本**几乎可忽略**（每用户每月增加不到 1 美分）
- 主要成本在**像素资产制作**和**固件开发**
- 如果用 AI 辅助生成像素资产，可进一步压缩

## 九、分阶段实施建议

### Phase 1: MVP（1-2 周）
- 10 种建筑类型 x 2 变体 = 20 个资产
- 固定 5x5 网格
- AI 决策用简单规则（完成率 -> 建筑类型映射）+ 随机变体
- iOS 无预览，仅 BLE 发送

### Phase 2: AI 增强（+1 周）
- 接入 OpenAI 做真正的 AI 创意决策
- 增加"建筑故事"文案（AI 生成每个建筑的小故事）
- iOS 端增加小镇预览 View

### Phase 3: 丰富内容（持续）
- 扩充到 25+ 建筑类型
- 季节/天气主题
- 特殊成就建筑
- 建筑互动（旋钮选中查看故事）

## 十、总结

| 维度 | 评估 |
|------|------|
| 技术可行性 | **高** — 完全契合现有"参数指令 -> 固件渲染"架构 |
| AI 成本增量 | **极低** — 每用户每月 <$0.01 |
| 主要投入 | **固件开发 + 像素资产** — 这是核心成本 |
| 用户体验价值 | **高** — 每天打开设备看到小镇变化，有持续惊喜感 |
| 风险 | **低** — 不影响现有功能，纯增量 |
