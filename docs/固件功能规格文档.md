## 1. 产品概述

### 1.1 产品定位

**Unlock the Flow, Make it Happen.**

Kirole 是一款面向深度知识工具用户的专注力伴侣设备。它把 iOS App 中的任务、日程、智能提醒、专注状态和三位 IP 伴侣同步到 E-ink 屏幕，帮助用户在低干扰界面里看到今天要做什么、现在该关注什么。

### 1.2 核心功能

| 功能 | 描述 |
|------|------|
| 伴侣陪伴 | Joy / Silas / Nova 三位 IP 伴侣提供短句陪伴和情绪反馈 |
| 专注反馈 | 反映用户专注模式和注意力数据，量化专注时间与中断次数 |
| 智能提醒 | 基于行为模式的上下文感知提醒，在合适时机推送到设备 |
| 任务展示 | 显示今日任务列表，支持完成标记 |
| 场景解锁 | 通过专注获得能量瓶，逐步解锁 harbor / forest / nightCity |
| 日程提醒 | 展示日历事件和时间安排 |
| 每日结算 | 统计当日完成情况，含专注指标和积分奖励 |

### 1.3 产品形态

- **设备类型:** 展示型 E-ink 设备
- **尺寸规格:** 4寸 / 7.3寸 两种规格
- **交互方式:** 电源键 + 旋钮按键 + BLE 连接 iOS App
- **供电方式:** 内置电池 + USB-C 充电

### 1.4 目标用户

- 深度知识工具用户（使用多种生产力工具但执行力不足）
- 远程工作者
- 自由职业者
- 需要专注力管理的用户
- 喜欢像素风格/电子宠物的用户

## 3. 显示页面设计

设备共有 4 个主要页面，通过时段自动切换或用户手动操作切换。**各页面的视觉布局/示意图以硬件团队的实际 UI 设计为准，本文档不再附 ASCII 示意图**；字段语义、触发与交互说明仍保留。

### 3.1 Page 1: Start of Day (早安页 / 每日开始日程概览)

**触发（v1.5.11，客户《电子墨水屏需求》调整）:** 两个条件**同时满足**时显示：
1. 日程表**全部日程未到达开始时间**（本地 RTC 对比 `Events[]` 各条 `Time`）；
2. **当日尚未进入过任务**（判定信号：本地未发过 `EnterTaskIn(0x10)`，且未收到过 Phase≠0 的 `0x14 FocusStatus`——App 侧发起的专注也计入「已进入任务」）。

开机默认显示行为保留（开机时以上两条件通常天然成立）。

**退出（三选一，先到先触发）:**
- 显示满 **10 分钟**自动切换 → Overview；
- **到达下一个日程的开始时间** → Overview；
- 任意按键 → Overview。

**数据字段:**

Page 1 与 Page 2 同属 `DayPack(0x10)`（见 §3.2 + BLE通信协议规格文档 §4.7）。一天开始页用到：`PetDialogue`（晨安气泡，单句）、`DaySummary`（一天总结，框②，**中性**面板文案——v2.5.30 起规则：日程繁忙/紧凑给休息建议，空闲则提醒喝水）、`FirstUp`（下一项，框③）、天气（`Weather 0x04`，含高/低温）。**旧字段 `morningGreeting / dailySummary / firstItem` 已于 v2.5.0 删除/更名，勿照其实现。**

**日程概览时间轴（v1.5.11 新增，固件渲染）:** 在 `DaySummary` 框下方绘制一条**横向时间轴**，把今日日程以**类别图标**打标在对应时间位置：
- 图标 = `Events[]` 每条的 `Category` 字节选择固件内置六类图标（BLE §4.7 Category 表，资产 `docs/assets/event-category-icons/`）；
- 位置 = 各条 `Time`（起始时间）在时间轴上的投影；相邻/重叠时段的图标可聚簇堆叠（参考客户示意图）；
- 端点 = 左太阳 / 右月亮（固件内置装饰图标）；首尾事件下方标注其时刻（如 "7:00" / "17:30"，来自 `Time`/`EndTime`）；
- 数据全部来自现有 DayPack 字段，**App 不为时间轴新增字段**。

### 3.2 Page 2: Overview (概览页)

> **重要（v2.5.0+，本节模型已更新）**：Page 1（一天开始）与 Page 2（概览）由**同一个 `DayPack(0x10)` 帧**驱动，不是各自独立的帧/字段。设备 UI **不是旧的「混合时间线」**（日程+任务交织成一条列表），而是「**常驻宠物气泡 + 可换数据面板**」：宠物口吻只有一句气泡 `PetDialogue`，其余皆为结构化面板数据（**事件卡、任务表分块，非混合**）。**权威字段与布局以 BLE通信协议规格文档 §4.7（DayPack）+ §6 / §6.5 为准**；下方旧 ASCII 示意与字段名仅作历史参考。

**面板构成（按设计稿渲染，详见 BLE §4.7 / §6.5）:**
- 顶栏: 天气（`Weather 0x04`，含高/低温）+ 日期。
- 宠物气泡: `PetDialogue`（单句，阶段感知）。
- 一天总结（框②）: `DaySummary`（**中性**面板文案，非宠物口吻；只谈日程 + 一条实用建议）。
- 下一项（框③）: `FirstUp`（下一个未来事件「HH:mm 标题」，无则置顶任务标题；App 算好下发）。
- 事件卡: `Events[]`（时间 / 标题 / 描述），按时间、只读。
- 任务表: `TopTasks[]`（标题 / 完成态 / 优先级），**按优先级排序**，可选进 Task In。
- 进度 / 能量: 来自 `SettlementData` 数值（完成 / 总数、积分、专注指标、能量瓶）。

> **双布局拍板（v1.5.11，客户《电子墨水屏需求》，替代旧「进度条 vs 任务清单」开放问题）**：概览页按**相邻日程间隔**在固件本地二分——取「前一个日程的 `EndTime` 到下一个日程的 `Time`」（"HH:mm" 本地转分钟比较，`EndTime` 为 BLE v2.5.30 新增字段）：
> - **情况一（间隔 <10 分钟）**：显示**当日日程完成情况比例**（进度条，`SettlementData.TasksCompleted/TasksTotal`，如 "●●●●●○○○○○ 50%"）+ 事件卡（`Events[]`）；
> - **情况二（间隔 ≥10 分钟）**：显示下一个日程的事件卡 + **任务清单**（`TopTasks[]`，"Task List" bullet 布局）。**恰好 10 分钟归情况二**（客户确认默认口径，2026-07-20，v1.5.12）。
>
> 判定在固件本地做，App 两组数据都发、**不引入 PanelMode 字节**（BLE v2.5.15 拍板不变，判定优先级表见 **BLE §6.7**：专注中→态 C 等）。「任务前三按 AI 排序」仍是**待客户确认项（暂缓）**，当前 App 按优先级排。

**显示时机:** 从 Start of Day 达到切换条件（10 分钟 / 到点 / 按键，见 §3.1）进入；或从专注任务页短按/长按退出后回到本页

**排序规则:**
- 日程: 按时间排序，仅显示不可交互
- 任务: 按优先级排序，可选择进入 Task In

**显示数量:** 根据屏幕尺寸调整 (4寸/7寸不同)
**空状态:** 显示"今日无任务"提示

**交互:**
- 滚轮滚动: 选择项目 (反色高亮)
- 按下滚轮/短按 (任务): 发送 `EnterTaskIn(0x10)`，等待 App 回发 `TaskInPage(0x11)` 后进入专注任务页
- 按下滚轮 (日程): 显示日程详情 (自动超时返回)
- **长按 (>1s)（v1.5.11 新增）: 「完成当日」→ 进入每日总结页（§3.4）**。与 Task In 页「长按=跳过任务」是**不同页面语境**，不冲突；误触保护见 §3.4

**数据字段:** 见 `DayPack(0x10)`，BLE通信协议规格文档 §4.7（`PetDialogue` / `Events[]`（每条含 `Time`/`EndTime`/`Category`）/ `TopTasks[]` / `SettlementData` / `DaySummary` / `FirstUp` / `SettlementReview` / `SettlementQuote`，后两者为 v2.5.30/v2.5.31 每日总结页文案）。**旧字段 `currentScheduleSummary / mixedTimeline / companionPhrase` 已于 v2.5.0 删除，勿照其实现。**

### 3.3 Page 3: Task In (任务详情页)

**显示时机:** 从 Overview 选择任务进入


**触发:** 从 Overview 选择任务进入
**交互:**
- 短按滚轮: 标记完成 → 自动返回 Overview
- 长按滚轮 (>1s): 跳过任务 → 返回 Overview

**事件同步:**
- 进入任务时: 发送 `EnterTaskIn` 事件 (带时间戳)
- 完成任务时: 发送 `CompleteTask` 事件 (带时间戳)
- 跳过任务时: 发送 `SkipTask` 事件 (带时间戳)

**数据字段:**

| 字段 | 最大长度 | 描述 |
|------|----------|------|
| taskTitle | 40 字符 | 任务标题 |
| taskDescription | 100 字符 | 任务描述 (可选，"Overview" 段) |
| encouragement | 50 字符 | ~~鼓励语（"*Tips" 段）~~ **v1.5.12 客户拍板停用：App 恒发空串，固件收到空串不渲染该行**（字段保留占位） |
| focusChallengeActive | 1 byte | 专注挑战状态 |

**三阶段画面（v1.5.11，客户《电子墨水屏需求》页面三）:** 专注页按已专注时长逐步切换三套画面（美术为固件内置，宠物姿态/瓶子填充渐进变化）：

| 阶段 | 时长 | 数据信号 |
|------|------|----------|
| 阶段一 | 0-5 分钟 | `0x14 FocusStatus` Phase=1 (warmup) |
| 阶段二 | 5-15 分钟 | Phase=2 (building) |
| 阶段三 | >15 分钟 | Phase=3 (deep) |

阶段以 `0x14` 的 `Phase` 字节为准（按**当前未打断段**计，打断退回 warmup，见 BLE §4.11）；**会话活跃时 Phase 恒 ≥1**（App v2.5.31 起第 0 分钟/打断清零瞬间也发 warmup，`Phase=0` 只表示无会话）；能量瓶子随 `Bottles` 字段逐步增多（App 侧显示封顶 5，v2.5.29）。

> **自定义 IP 在专注页（v1.5.12）**：自定义形象激活时，专注页**仍用内置 IP 美术**（按最近一次 `0x01 CharacterId`）——用户单图无三阶段姿态，多图上传暂不支持（客户 2026-07-20 确认现状；其余页面见 §5.1 口径）。

### 3.4 Page 4: Settlement (结算页 / 每日总结)

**触发（v1.5.11，客户《电子墨水屏需求》调整）:** 概览页**长按按钮「完成当日」**进入（替代旧「所有任务完成自动显示」——自动触发已废弃）。

**误触保护 / 退出:** 在本页**再次长按** → 返回概览页（客户明确的防误触设计）。

**内容两段（v2.5.30/v2.5.31 DayPack 文案字段，见 BLE §4.7 / §6.4；客户确认"分3部分"系笔误）:**

### 3.6 屏保模式

**触发:** 按电源键
**内容:** 静态伴侣场景图（桌面装饰，满屏 1600×1200，底图**无文字**）+ **金句文本叠加层**
**金句:** 由 App 生成、经 BLE 推送的**动态文本**（如励志引文 + 作者），固件运行时叠加绘制在屏保图上（建议绘制区＝画面上半中部）。屏保底图本身不含任何文字。
- **帧已升级为业务帧（v2.5.10）**：金句经 **`0x16` Screensaver 业务帧** 下发（payload 字段 `ContentType / SceneByte / PostcardDay / QuoteLen+Quote / AuthorLen+Author`，见 BLE 通信协议规格文档 **§4.15**）。App 端已实现金句**生成**（`ScreensaverService.getScreensaverConfig` / `OpenAIService.generateScreensaverQuote`，引文 <60 字符）与**封帧下发**（`BLEDataEncoder.encodeScreensaver` → `BLEService.sendScreensaverConfig`，经 `writeData`）。
- ⚠️ **发送路径（固件须知）**：
  1. **被动触发、且依赖此刻仍连接**：App 不主动定时推送金句，而是在收到硬件 sleep 事件（`handleHardwareSleep`）后即时下发。若固件进入屏保后**立刻断连**，这帧可能来不及送达——固件应在断连前预留收帧窗口，或缓存上一次金句。
  2. ~~secure mode 下被拒~~ **已解决（v2.5.10）**：金句已从旧 `0xAA 01 02` 开发命令升级为 `0x16` 业务帧、改走 `writeData`——**dev / secure（生产签名通道）两种模式均可发**（secure 下自动经 SecureEnvelope 封装）。原「仅开发/未签名模式可达」的限制不再存在。
- ⚠️ **渲染侧（固件待实现）**：**固件需正式支持 `0x16` Screensaver 业务帧并渲染 Quote/Author**（按 BLE 协议 §4.15 的 payload 布局解析）。这是标准业务帧、不再是可忽略的 `0xAA` 开发命令。屏保帧**可能单包亦可能分包**（Quote/Author 长或 secure 封装后超 MTU），固件须按 §3.2 通用分包重组、不可假设恒为单包。
- ⚠️ **场景底图经 `0x17` SceneUnlock 切换（固件待实现）**：屏保叠加金句的**底图**（harbor/forest/nightCity）由 App 经 `0x17` SceneUnlock 业务帧推送/切换（payload = 1B SceneId：`0x00=harbor / 0x01=forest / 0x02=nightCity`，恒为单包，见 BLE 协议 §4.16）。旧 `0xAA 01 01` 场景命令已废弃；注意 `0x17` 出站=SceneUnlock、入站=ReminderDismissed（方向分离）。
**BLE:** 进入屏保后断开连接（省电）——因此金句须在**断连前**送达并由固件缓存，断连期间用缓存文本渲染。
**退出（v1.5.11 调整）:** 再次短按电源键 → **切回进入屏保前的页面**（客户《电子墨水屏需求》页面四「跳出条件：再次短按电源键，切回原界面」；原「返回 Start of Day」废弃）

## 5. 伴侣显示系统

### 5.1 伴侣角色

当前产品只有三位 IP 伴侣，设备通过 `PetStatus.CharacterId` 区分：

| CharacterId | 显示名 | 固件含义 |
|-------------|--------|----------|
| `joy` | Joy | 喜乐，轻松陪伴 |
| `silas` | Silas | 仁爱，温和稳住用户 |
| `nova` | Nova | 节制 / 自律，冷静聚焦 |

固件不得再实现旧的多形态选择系统。角色身份以 App 下发的 `CharacterId` 为准。

> **自定义 IP（v1.5.12/v1.5.13，客户 2026-07-20 口径 + CustomActive 字节）**：用户可在 App 内创建自定义形象，经 `0x15 CustomAvatarFrame`（PNG 单图，BLE §4.12）下发，**固件须持久化该图**（重启/重连不丢）。是否处于自定义显示模式由 `0x01 PetStatus` 的 `CustomActive` 字节（BLE §4.2，v2.5.32）**显式判定**：`=1` → 除专注页外的全部页面（早安/概览/每日总结/屏保等）显示用户图、不随任务/标签切换，且本帧不触发内置形象重绘；`=0` → 按 `CharacterId` 渲染内置。**专注页恒用内置 IP 美术**（按 `CharacterId`，用户图无三阶段姿态）。`CustomActive=1` 但本地尚无 0x15 图时暂按 CharacterId 内置渲染、图到即切。**v1.5.14 生命周期补充**：删除激活中的自定义形象时 App 会立即发 `0x01`（CustomActive=0），固件切回内置；换设备场景由 App 自动重推 `0x15`（记录上次收图设备）；**同一台设备持久化图被清空**是当前已知盲区——App 无法感知，建议固件评估在 `DeviceWake(0x30)` 追加"本地无自定义图"标志位（1 bit）以闭环，未定稿前用户可在 App 重选自定义形象手动恢复。

## 6. 数据同步

### 6.1 BLE 协议概要

第一次硬件联调请先参考 `BLE初次联调指南.md`。完整协议请参考 `BLE通信协议规格文档.md`。

BLE `Type` 字节必须按方向解释。同一个 `Type` 在 App→Device 和 Device→App 中可以有不同含义，固件不能只按 `Type` 判断业务。

#### 主要命令

| 命令 | 类型码 | 方向 | 描述 |
|------|--------|------|------|
| DayPack | 0x10 | App → Device | 发送每日数据包 |
| TaskInPage | 0x11 | App → Device | 发送任务详情 |
| DeviceMode | 0x12 | App → Device | 设置设备模式 |
| SmartReminder | 0x13 | App → Device | AI 智能提醒推送 |
| FocusStatus | 0x14 | App → Device | 推送当前专注阶段、会话能量瓶子数、已专注分钟数和任务标题 |
| EventLogRequest | 0x20 | App → Device | 请求指定时间戳之后的事件日志 |
| DeviceWake | 0x30 | Device → App | 设备唤醒并上报电量 |
| EnterTaskIn | 0x10 | Device → App | 进入任务详情，触发 App 回发 TaskInPage |
| CompleteTask | 0x11 | Device → App | 任务完成事件 |
| WheelSelect | 0x14 | Device → App | 滚轮选择确认 (发送选中项 ID；App 当前仅记录/调试，不回发详情) |
| ViewEventDetail | 0x15 | Device → App | 查看日程详情 |
| ReminderAcknowledged | 0x16 | Device → App | 用户确认收到提醒 |
| ReminderDismissed | 0x17 | Device → App | 提醒超时自动关闭 |
| RequestRefresh | 0x20 | Device → App | 请求刷新数据 |
| EventLogBatch | 0x21 | Device → App | 批量回传事件日志 |
| LowBattery | 0x40 | Device → App | 低电量通知 |

> 说明：`0x10`、`0x11`、`0x12`、`0x13`、`0x14`、`0x20`、`0x21` 都可能按方向复用。例如 App→Device 的 `0x14` 是 `FocusStatus`，Device→App 的 `0x14` 是 `WheelSelect`。固件需按链路方向区分解析。

### 6.2 事件发送时机

| 触发条件 | 发送事件 |
|----------|----------|
| 滚轮滚动 | 纯本地处理，不发送事件 |
| 按下确认并进入任务详情 | EnterTaskIn (发送 TaskId + 时间戳；App 回发 TaskInPage) |
| 普通确认但不进入任务详情 | WheelSelect (发送选中项 ID；App 当前仅记录/调试，不回发详情) |
| 按下确认 (日程) | ViewEventDetail |
| 完成任务 | CompleteTask |
| 跳过任务 | SkipTask |
| 设备唤醒 | DeviceWake (发送 BatteryLevel) |
| 用户手动刷新 | RequestRefresh |
| 用户按键关闭提醒 | ReminderAcknowledged (0x16) |
| 提醒 10 秒超时 | ReminderDismissed (0x17) |
| 电量低于 20% | LowBattery |

第一轮联调中，任务详情页可以暂不实现；如果暂不实现，不要用 `WheelSelect` 代替 `EnterTaskIn`，否则 App 不会回发 TaskInPage。

### 6.3 数据刷新时机

| 触发条件 | 刷新内容 |
|----------|----------|
| App 启动 | 完整 DayPack |
| 设备连接 | 完整 DayPack |
| 任务状态变更 | 增量更新 |
| 用户手动刷新 | 完整 DayPack |
| 定时刷新 (每小时) | 完整 DayPack |

### 6.4 初次联调范围

第一次实验室联调只验证 BLE 基础收发：

- 广播 `FFE0` Service UUID。
- 提供 `FFE1` Write 和 `FFE2` Notify。
- 发送 `DeviceWake(0x30)` 上报电量。
- 接收并打印 `Time(0x05)`。
- 发送 `RequestRefresh(0x20)`。
- 接收并重组 `DayPack(0x10)`，先打印字段，不要求完整 UI 渲染。

第一次联调暂不验收：

- 安全握手和 `SecureData(0x7E)`。
- `EventLogBatch(0x21)` 离线补传。
- `TaskInPage(0x11)` 详情页。
- `FocusStatus(0x14)` 专注实时状态。
- `SmartReminder(0x13)`。
- Spectra 6 图片帧传输。

### 6.5 离线行为

| 场景 | 设备行为 |
|------|----------|
| 未连接 App | 显示缓存数据 |
| 按键完成任务 | 本地标记 + 延迟同步 |
| 跨日未同步 | App 推送新 Day Pack |

### 7.2 字体（字库 / 角色映射 / 字号）

**内置两套字库**（已烧入固件，App **不传字体**——按字段语义角色渲染，见 BLE 协议 §6.6）：

| 字体 | 角色 | 用于 |
|------|------|------|
| **Lugrasimo**（手写 / 装饰） | 宠物口吻 | `PetDialogue` 伴侣气泡（**仅此一处**） |
| **Calibri**（无衬线 / 数据） | 中性·数据 | 顶栏天气/日期、进度/结算、事件卡（标题+描述）、`DaySummary`、`TaskInPage`、屏保 Quote/Author 等其余**所有**文本 |

> **字形回退（兜底安全网，非常态）**：自 BLE 协议 v2.5.14 起，App 已在编码边界把**所有**出站文本净化为纯可打印 ASCII（`0x20`–`0x7E`，见 BLE 协议 §3.5）——`PetDialogue` 里原先偶发的弯引号 `’ “ ”` / 长破折号 `—` / emoji **不再到达固件**（App 侧已转直引号 `'` / `-`、emoji 丢弃）。故 **Lugrasimo 只需覆盖可打印 ASCII（`0x20`–`0x7E`）字形**；万一某 ASCII 字形仍缺失，**回退 Calibri**（标准 glyph fallback，此时仅兜底、正常不触发）。**不逐串传字体**：字库已烧进固件，按「字段角色 → 字体」映射即可。

## 8. 固件开发指南

### 8.1 开发环境

| 项目 | 推荐 |
|------|------|
| MCU | ESP32-S3 模组 |
| IDE | ESP-IDF v5.x |
| E-ink 驱动 | GxEPD2 / 厂商 SDK |

### 8.2 内存布局

```
Flash (>= 16MB SPI NOR):
├── Bootloader (64KB)
├── Firmware (2MB)
├── OTA 分区 (2MB, 可选)
├── Companion Assets (3MB)
│   ├── joy/
│   ├── silas/
│   └── nova/
├── Fonts / 字库 (1MB)
├── 屏保图片 (4寸 120KB / 7寸 960KB)   ⚠️ 7寸 1600×1200 全屏屏保单张 960KB，原 512KB 不足
├── 彩色页面缓存 (>= 4 页)
│   ├── 4寸: 4 × 120KB = 480KB
│   └── 7.3寸: 4 × 960KB = 3.84MB   ⚠️ 1600×1200，需复核 Flash 余量
├── Event Log 环形缓冲 (256KB)
└── 预留 (7寸方案约 2.9MB)   ⚠️ 按实物 1600×1200：页缓存 3.84MB、屏保 960KB，预留偏紧，请确认 16MB Flash 够用
```

PSRAM 用途 (>= 2MB，推荐 4MB):
- 屏幕帧缓冲: 4寸 120,000 bytes / **7.3寸 960,000 bytes** (Spectra 6 4bpp) ⚠️ 7.3寸 1600×1200，单帧 960KB，PSRAM 余量需复核
- BLE 分包缓存 (包头 11 字节 + payload，需支持多包重组；v2.5.24 前为 9 字节。0x15 头像 PNG 重组缓冲最坏 ≤1MiB，请计入 PSRAM 预算)
- 0x15 头像 PNG **解码**缓冲：800×700 RGBA 位图约 **2.24MB** + zlib inflate 工作集（App 保证 8-bit sRGB、可能含 alpha）。与重组缓冲（≤1MiB）、帧缓冲（7.3 寸 960KB）叠加时请复核 PSRAM 余量；如紧张可流式解码按行转换 4bpp、不落全幅 RGBA
- UI 临时数据

### 8.3 关键实现点

#### 电源管理

```c
// 休眠唤醒
void enter_deep_sleep() {
    // 保存当前状态
    save_state_to_rtc();
    // 配置唤醒源: 按键 + 定时器
    esp_sleep_enable_ext0_wakeup(BUTTON_PIN, 0);
    esp_sleep_enable_timer_wakeup(SLEEP_DURATION_US);
    esp_deep_sleep_start();
}
```

#### E-ink 刷新策略

```c
typedef enum {
    REFRESH_FULL,      // 全刷新 - 页面切换
    REFRESH_PARTIAL,   // 局部刷新 - 任务状态更新
    REFRESH_FAST       // 快速刷新 - 动画帧
} RefreshMode;
```

#### BLE 数据接收

简单命令使用 3 字节头 (type + length)，大数据使用 **11 字节分包头**（v2.5.24 起，`Seq`/`Total` 各 2B BE；详见 BLE通信协议规格文档 Section 3.2）。

```c
void on_ble_data_received(uint8_t* data, size_t len) {
    if (len < 3) {
        return;
    }

    uint8_t type = data[0];
    uint16_t payload_len = (data[1] << 8) | data[2];
    if (len != 3 + payload_len) {
        return;
    }

    uint8_t* payload = &data[3];

    switch (type) {
        case 0x10: // DayPack (分包重组后)
            parse_day_pack(payload, payload_len);
            refresh_display(REFRESH_FULL);
            break;
        case 0x11: // TaskInPage
            parse_task_in_page(payload, payload_len);
            show_task_in_page();
            break;
        case 0x13: // SmartReminder
            parse_smart_reminder(payload, payload_len);
            show_reminder_banner();
            break;
        // ...
    }
}
```

DayPack / TaskInPage / CustomAvatarFrame 可能以 **11 字节分包**到达（v2.5.24）。固件应先按 `MessageId + Seq + Total` 重组（Seq/Total 各 2B BE，总数上限 65535），校验每包 CRC16-CCITT-FALSE，再把完整 payload 交给页面解析；头像帧长传输期间其他帧可能以不同 MessageId 交错到达，重组须按 MessageId 分槽。当前 App 不支持分包 ACK，也不支持单包重传；固件发现缺包、CRC 错或超时时，丢弃整条消息，等待 App 后续重新发送。
